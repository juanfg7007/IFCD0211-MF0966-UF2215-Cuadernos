UF2215. UD1. Actividad 5. Ejercicio teórico-práctico: Control de concurrencia


Enunciado (Descripción)

 Supongamos un caso práctico donde se producen transacciones bancarias y se deposita una cantidad de dinero en una cuenta que contiene X=3.000 €. El control de transacciones concurrentes en la base de datos del banco brinda un eficiente desempeño del Sistema de Administración de Base de Datos, puesto que permite controlar la ejecución de transacciones que operan en paralelo, accediendo a información compartida y, por lo tanto, interfiriendo potencialmente unas con otras.

Las transacciones son las siguientes:

Tenemos una transacción T1 que realiza un Cliente1:

Lectura(X)
X: =X+100
Escritura(X) 

Y una transacción T2 que realiza un Cliente2:

 Lectura(X)
X:= X+100000
Escritura (X)

T1 T2 X
Lectura(X)

X:=X+100


Escritura(X) 		 Lectura(X)

X:=X+100000
Escritura(X)		 X
X
X
X
X+100
X+100


Preguntas / Actividades a realizar 

Pregunta 1A. Si dichas transacciones se ejecutan simultáneamente, ¿cuál sería el resultado que obtendríamos? ¿Qué problema observamos?

Pregunta 1B. ¿De cuántos algoritmos de control de concurrencia disponemos? Explica detalladamente cada uno de ellos.

Pregunta 1A

Si las transacciones T1 y T2 se ejecutan simultáneamente sin un control de concurrencia adecuado, podríamos obtener resultados no deseados debido a la interferencia entre las transacciones.

El problema principal es la interferencia o conflicto entre las operaciones de lectura y escritura de las transacciones T1 y T2. Ambas transacciones leen el valor actual de X, realizan operaciones y luego escriben el nuevo valor. Si estas operaciones se ejecutan simultáneamente sin un mecanismo de control de concurrencia, podríamos tener resultados incorrectos debido a la superposición de las operaciones.

En el caso específico:

  • Ambas transacciones leen el valor original de X (3000 €) simultáneamente.
  • La transacción T1 suma 100 al valor leído (3000 + 100 = 3100) y escribe el nuevo valor en X.
  • Mientras tanto, la transacción T2 suma 100,000 al valor leído originalmente (3000 + 100,000 = 103,000) y escribe este nuevo valor en X.

Como resultado, el valor final de X dependerá de la secuencia exacta de ejecución de las operaciones. Podríamos obtener X=3100 si T1 se ejecuta primero, o X=103,000 si T2 se ejecuta primero.

Este escenario ilustra el problema de la inconsistencia debido a la falta de control de concurrencia. En entornos con transacciones concurrentes, es esencial implementar mecanismos como el bloqueo de recursos compartidos o la gestión de concurrencia mediante sellos temporales para evitar resultados indeseados. Sin estos mecanismos, las transacciones pueden interferirse entre sí, llevando a resultados impredecibles y violando la integridad de la base de datos. Implementar un control de concurrencia adecuado garantiza la consistencia de los datos y evita conflictos en entornos con múltiples transacciones concurrentes.

Pregunta 1B

Existen varios algoritmos de control de concurrencia utilizados en sistemas de gestión de bases de datos (SGBD):

  • Bloqueo (Locking)
    • Este enfoque implica el uso de bloqueos para gestionar el acceso a los recursos compartidos, como registros o páginas de datos.
    • Funcionamiento: cuando una transacción accede a un recurso, adquiere un bloqueo sobre ese recurso, lo que impide que otras transacciones accedan al mismo recurso simultáneamente.
    • Tipos de bloqueo: existen bloqueos compartidos y bloqueos exclusivos. Los bloqueos compartidos permiten a varias transacciones leer un recurso simultáneamente, pero impiden las escrituras. Los bloqueos exclusivos evitan tanto lecturas como escrituras concurrentes
    • Ventajas: fácil implementación, control preciso sobre el acceso a recursos.
    • Desventajas: puede llevar a bloqueos mutuos (deadlocks) si no se gestiona adecuadamente.

  • Sellos Temporales (Timestamping)
    • Cada transacción y recurso tienen asociados sellos temporales. Se utiliza para ordenar las transacciones y gestionar la concurrencia.
    • Funcionamiento: cada vez que una transacción desea acceder a un recurso, se le asigna un sello temporal. Se compara con los sellos temporales de otras transacciones y se toman decisiones basadas en estas comparaciones.
    • Conflictos: se resuelven conflictos mediante reglas como "la transacción con el sello temporal más antiguo tiene prioridad".
    • Ventajas: evita bloqueos mutuos, proporciona un ordenamiento claro de las transacciones.
    • Desventajas: requiere la generación y gestión de sellos temporales, puede haber conflictos de concurrencia.

  • Control de Concurrencia Multiversión (Multiversion Concurrency Control - MVCC)
    • Cada transacción ve una "versión" consistente de la base de datos según el momento en que comenzó.
    • Funcionamiento: se crean múltiples versiones de cada objeto en la base de datos, cada una con un sello temporal. Las transacciones leen la versión que existía en el momento de su inicio.
    • Ventajas: alta concurrencia, evita bloqueos, cada transacción ve una vista consistente de la base de datos.
    • Desventajas: puede aumentar la complejidad del sistema, requerir más espacio de almacenamiento para versiones múltiples.
